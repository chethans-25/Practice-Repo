Here is a comprehensive test case for your ARINC 429 VIP (Verification IP) to verify the Null state timeout requirement.
​Because ARINC 429 uses Bipolar Return-to-Zero (BPRZ) encoding, a standard gap between 32-bit words is typically 4 bit times. This requirement ensures your Receiver VIP correctly identifies when the bus goes completely idle beyond the standard inter-word gap.
​
Test Case: ARINC_429_RX_NULL_TIMEOUT_001
​
Objective: Verify that the Receiver VIP identifies and logs a "Null (no data)" condition exclusively when the bus remains in the Null state (0V differential) for strictly greater than 5 bit times.
​
Prerequisites:
​The ARINC 429 Receiver VIP is configured and instantiated in the testbench.
​The bus speed is configured (e.g., High Speed at 100 kbps where 1 bit time = 10 µs, or Low Speed at 12.5 kbps where 1 bit time = 80 µs).
​The bus is initialized to the Null state.
​
Stimulus and Observation Sequence

Step; Stimulus Action (TX/Driver)                                                               Observation Point (RX/Monitor)
1.    Transmit a valid 32-bit ARINC 429 word (e.g., Label 055).                                 RX VIP successfully receives and decodes the 32-bit word.
2.    Drive the bus to the Null state for exactly 4.0 bit times.                                No Null condition logged. This is a standard inter-word gap.
3.    Transmit a second valid 32-bit ARINC 429 word.                                            RX VIP successfully receives the second word.
4.    Drive the bus to the Null state for exactly 5.0 bit times.No Null condition logged.       Boundary condition is respected.
5.    Transmit a third valid 32-bit ARINC 429 word.                                             RX VIP successfully receives the third word.
6.    Drive the bus to the Null state and hold it for > 5.0 bit times (e.g., 6.0 bit times).    CRITICAL CHECK: At exactly 5+ bit times (plus any clock sampling delta), the VIP logs the "Null (no data)" condition.
7.    Transmit a fourth valid 32-bit ARINC 429 word.                                            VIP recovers gracefully from the Null state, clears the condition, and receives the word.

Verification IP Implementation Details
​To make this test robust in a standard UVM/SystemVerilog environment, you should track the following hooks in your VIP:
​
1. Boundary Value Analysis (BVA)
​Hardware clocks and sampling can introduce edge-case race conditions. Run this sequence with randomized timing sweeps at the boundaries:
​Sweep 1: 4.99 bit times (Must NOT trigger)
​Sweep 2: 5.00 bit times (Must NOT trigger)
​Sweep 3: 5.01 bit times (MUST trigger)
​
2. Assertions (SVA)
​If your VIP includes SystemVerilog Assertions, you can bind an assertion to the receiver interface to catch this automatically without relying purely on UVM logs.
​An example property logic:
// Assuming 'rx_null' is a signal indicating the bus is in the Null state
// and 'bit_clk' is the recovered or reference bit clock.

property p_null_timeout;
  @(posedge bit_clk) 
  $rose(rx_null) |-> (rx_null [*5]) |=> rx_null_logged_flag;
endproperty

assert_null_timeout: assert property(p_null_timeout) 
  else $error("Receiver failed to identify Null state over 5 bit times.");


Log Checking
​Your testbench scoreboard or sequence must parse the VIP's logger output (e.g., via uvm_info or uvm_warning). Verify that the text logged exactly matches the required terminology: "Null( no data )".
